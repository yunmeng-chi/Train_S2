### 【漏洞 1】LFI 目录穿越读取任意文件（CWE-22/CWE-98）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name={path}` |
| **原始风险等级** | 🔴 **紧急** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-22: Improper Limitation of a Pathname to a Restricted Directory / CWE-98: Improper Control of Filename |

#### 漏洞形成原理（修复前）

`/page` 路由直接将用户输入的 `name` 参数拼接到文件路径中，**未做任何过滤**：

```python
# 修复前漏洞代码
name = request.args.get("name", "")           # ← 用户完全控制
filepath = os.path.join("pages", name)        # ← 无 ../ 检测
with open(filepath, "r", encoding="utf-8") as f:
    content = f.read()                        # ← 任意文件读取
```

**攻击示例**：
```
GET /page?name=../../root/.ssh/id_rsa     → 读取 SSH 私钥
GET /page?name=../../etc/passwd           → 读取系统密码文件
GET /page?name=../../app.py              → 读取应用源码
GET /page?name=../data/users.db           → 下载 SQLite 数据库
GET /page?name=../../var/log/auth.log     → 读取 SSH 登录日志
```

#### 修复方案

采用 **白名单机制** + **路径规范化二次防护**双重防御：

```python
# 安全白名单：仅允许预定义的页面名
ALLOWED_PAGES = {"help", "about", "faq"}
PAGES_DIR = os.path.join(os.path.dirname(os.path.abspath(__file__)), "pages")

@app.route("/page")
def page():
    if not session.get("username"):
        return redirect("/login")

    name = request.args.get("name", "")

    # 【第一重防御】白名单校验——只允许预设页面
    if name not in ALLOWED_PAGES:
        return "页面不存在", 404

    # 【第二重防御】路径规范化 + 二次防护
    filename = name + ".html"
    filepath = os.path.join(PAGES_DIR, filename)
    real_path = os.path.realpath(filepath)

    # 确保最终路径在 pages 目录内
    if not real_path.startswith(os.path.realpath(PAGES_DIR) + os.sep):
        return "页面不存在", 403
```

#### 修复效果验证

| 测试用例 | 预期结果 | 实际结果 |
|---------|---------|:-------:|
| `/page?name=help`（正常） | 显示帮助中心 | ✅ 通过 |
| `/page?name=../../etc/passwd` | 返回 404 | ✅ 已阻断 |
| `/page?name=../../root/.ssh/id_rsa` | 返回 404 | ✅ 已阻断 |
| `/page?name=../data/users.db` | 返回 404 | ✅ 已阻断 |
| `/page?name=nonexistent` | 返回 404 | ✅ 通过 |

### 【漏洞 2】SSH 私钥泄露致服务器沦陷（CWE-522）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name=../../root/.ssh/id_rsa` |
| **原始风险等级** | 🔴 **紧急** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-522: Insufficiently Protected Credentials |

#### 漏洞形成原理（修复前）

应用以 root 权限运行，LFI 漏洞可直接读取 `/root/.ssh/id_rsa`。攻击者拿到私钥后 `ssh -i id_rsa root@<server_ip>` 即获得服务器 root shell。

#### 修复方案

该漏洞由漏洞 1（LFI 目录穿越）直接导致。**LFI 白名单修复后，所有目录穿越请求均被阻断**，SSH 私钥无法再通过 `/page` 接口读取。

#### 修复效果验证

| 测试用例 | 预期结果 | 实际结果 |
|---------|---------|:-------:|
| `curl /page?name=../../root/.ssh/id_rsa` | 不返回私钥内容 | ✅ 已阻断 |
| 检查私钥文件是否存在于 pages/ 目录 | 不存在 | ✅ 安全 |

### 【漏洞 3】路径编码绕过（CWE-174）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name={encoded_path}` |
| **原始风险等级** | 🟠 **高危** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-174: Double Decoding of the Same Data |

#### 漏洞形成原理（修复前）

Flask 自动对 URL 参数进行一次 URL 解码，攻击者可利用双层编码或双斜杠绕过简单的黑名单过滤：

| 绕过方式 | 攻击示例 |
|---------|---------|
| **双层 URL 编码** | `%252e%252e%252f` → Flask 解码为 `%2e%2e%2f` → open() 解码为 `../` |
| **双斜杠绕过** | `....//....//` → 等效于 `../../` |
| **绝对路径绕过** | `/etc/passwd` → `os.path.join("pages", "/etc/passwd")` → `/etc/passwd` |

#### 修复方案

白名单方案从根本上杜绝了所有编码绕过——**只有白名单内的名称会被处理，任何非白名单输入（包括编码后的路径穿越 payload）都返回 404**：

```python
# 白名单校验在 URL 解码后执行
name = request.args.get("name", "")
if name not in ALLOWED_PAGES:
    return "页面不存在", 404
```

同时增加 `os.path.realpath()` 二次防护，即使存在未知绕过也能确保路径被锁定在 `pages/` 目录内。

#### 修复效果验证

| 测试用例 | 预期结果 | 实际结果 |
|---------|---------|:-------:|
| `/page?name=%252e%252e%252fetc/passwd`（双层编码） | 返回 404 | ✅ 已阻断 |
| `/page?name=....//....//....//etc/passwd`（双斜杠） | 返回 404 | ✅ 已阻断 |
| `/page?name=/etc/passwd`（绝对路径） | 返回 404 | ✅ 已阻断 |

### 【漏洞 4】日志投毒 + LFI 远程代码执行（CWE-73/CWE-98）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | SSH 日志投毒 + `GET /page?name=../../var/log/auth.log` |
| **原始风险等级** | 🟠 **高危** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-73: External Control of File Name or Path |

#### 漏洞形成原理（修复前）

攻击者利用 SSH 日志注入恶意代码后通过 LFI 包含日志文件，使代码被执行：

```
Step 1: SSH 日志投毒
$ ssh "{{SSTI_PAYLOAD}}"@<target>
→ 认证失败记录写入 /var/log/auth.log

Step 2: LFI 包含日志
GET /page?name=../../var/log/auth.log
→ 日志内容被渲染 → Jinja2 SSTI 执行 → RCE
```

#### 修复方案

三重防御彻底阻断日志投毒攻击链：

```python
# [防御1] 白名单阻断目录穿越（漏洞1修复）
if name not in ALLOWED_PAGES:
    return "页面不存在", 404       # auth.log 被 404 阻断

# [防御2] 路径规范化防护（漏洞3修复）
real_path = os.path.realpath(filepath)
if not real_path.startswith(os.path.realpath(PAGES_DIR) + os.sep):
    return "页面不存在", 403

# [防御3] 输出清洗——即使文件被读取，也清洗恶意代码
content = bleach.clean(content, tags=allowed_tags, strip=True)
```

#### 修复效果验证

| 攻击环节 | 阻断点 | 效果 |
|---------|--------|:----:|
| 日志投毒写入 auth.log | SSH 认证日志（外部无法控制） | ✅ 外部无法写入系统日志 |
| LFI 包含日志 | 白名单阻断 `../../var/log/auth.log` | ✅ 已阻断 |
| 代码执行渲染 | Bleach 清洗移除恶意标签 | ✅ 已阻断 |

### 【漏洞 6】XSS 反射型 — page_content 无过滤渲染（CWE-79）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name={html_with_xss}` |
| **原始风险等级** | 🟡 **中危** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-79: Improper Neutralization of Input During Web Page Generation |

#### 漏洞形成原理（修复前）

`index.html` 使用 `{{ page_content | safe }}` 渲染页面内容，Jinja2 的 `| safe` 过滤器禁用 HTML 转义，恶意 HTML/JavaScript 被直接执行。

```html
<!-- 修复前：禁用转义，直接渲染原始 HTML -->
{{ page_content | safe }}

<!-- 攻击载荷：<script>alert(document.cookie)</script>
     → JS 被执行，窃取用户 Session -->
```

#### 修复方案

**双重防护**：后端 bleach 清洗 + 前端移除 `| safe`：

**修复 1 — 后端清洗（app.py）：**
```python
import bleach

# 只允许安全的 HTML 标签
allowed_tags = ["h2", "h3", "h4", "p", "b", "strong", "em", "i",
                "hr", "br", "ul", "ol", "li", "a", "code", "pre",
                "blockquote", "table", "thead", "tbody", "tr", "th", "td"]
allowed_attrs = {"a": ["href", "title"]}
content = bleach.clean(content, tags=allowed_tags, attributes=allowed_attrs, strip=True)
```

**修复 2 — 前端模板（index.html）：**
```html
<!-- 修复后：移除 | safe，使用 Jinja2 自动转义 -->
{% if page_content %}
<div class="card">
    <div class="page-content">
        {{ page_content }}
    </div>
</div>
{% endif %}
```

#### 修复效果验证

| 测试用例 | 预期结果 | 实际结果 |
|---------|---------|:-------:|
| `<script>alert(1)</script>` | 被删除 | ✅ `alert(1)`（文本） |
| `<img src=x onerror=alert(1)>` | 被删除 | ✅ 完全删除 |
| `<a href="javascript:alert(1)">click</a>` | 清理 href | ✅ `<a>click</a>` |
| `<h2>标题</h2><p>段落</p>` | 保留正常标签 | ✅ 正常显示 |

### 【漏洞 8】上传文件名路径穿越（CWE-22/CWE-434）

|------|------|

#### 漏洞形成原理（修复前）

虽然做了扩展名白名单校验，但**未对文件名中的路径穿越字符进行过滤**：

```python
# 修复前：直接使用用户提供的文件名
filepath = os.path.join("static/uploads", file.filename)  # ← ../../templates/evil.html
```

**攻击示例**：
```

→ 文件实际保存为 templates/evil.html → 覆盖 Jinja2 模板 → SSTI → RCE
```

```python
# 修复后：使用 UUID 重命名文件
import uuid

safe_filename = f"{uuid.uuid4().hex}.{ext}"     # 如 "a1b2c3d4e5f6...jpg"
filepath = os.path.join("static/uploads", safe_filename)
file.save(filepath)
```

### 【漏洞 9】上传文件内容未检测 — 图片马/WebShell（CWE-434）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `POST /upload` |
| **原始风险等级** | 🟠 **高危** |
| **当前状态** | ✅ **已修复** |
| **CWE** | CWE-434: Unrestricted Upload of File with Dangerous Type |

#### 漏洞形成原理（修复前）

仅校验扩展名白名单，**未检查文件真实内容**。攻击者可将恶意文件伪装为图片上传：

```
# 创建一个"图片马"——实际是 PHP Webshell 但后缀是 .jpg
echo '<?php system($_GET["cmd"]); ?>' > shell.jpg
# 上传后可通过路径穿越写入可执行目录
```

#### 修复方案

**Magic Number 文件幻数十进制校验**——验证文件真实类型：

```python
magic_bytes = file.read(16)
file.seek(0)

# 各种图片格式的 Magic Number
magic_map = {
    b"\xff\xd8": "jpg",           # JPEG
    b"\x89PNG\r\n\x1a\n": "png",  # PNG
    b"GIF87a": "gif",             # GIF
    b"GIF89a": "gif",             # GIF
    b"RIFF": "webp",              # WebP
    b"BM": "bmp",                 # BMP
}

detected_ext = None
for magic, mext in magic_map.items():
    if magic_bytes[:len(magic)] == magic:
        detected_ext = mext
        break

if detected_ext is None:
    return render_template("upload.html", error="文件内容不是有效的图片格式"), 400
```

#### 修复效果验证

| 测试用例 | 预期结果 | 实际结果 |
|---------|---------|:-------:|
| 真实 JPEG 文件（含 JFIF 头） | 上传成功 | ✅ 通过 |
| 真实 PNG 文件 | 上传成功 | ✅ 通过 |
| PHP Shell 文件后缀改为 .jpg | 拒绝上传 | ✅ 已阻断 |
| `<script>` 文本保存为 .png | 拒绝上传 | ✅ 已阻断 |

#### 漏洞形成原理（修复前）

Flask 默认将 `/static/` 路径映射为静态文件服务目录，所有上传文件可被匿名用户通过 URL 直接访问。虽然 UUID 重命名后文件名不可预测，但仍存在路径扫描风险。

#### 修复方案

添加安全 HTTP 响应头，并设置 `X-Content-Type-Options: nosniff` 防止 MIME 类型嗅探攻击：

```python
@app.after_request
def add_security_headers(response):
    """添加安全响应头"""
    response.headers["X-Content-Type-Options"] = "nosniff"
    return response
```

## 六、伪协议拓展配置分析

### 6.1 伪协议兼容性总览

`/page` 路由使用 Python 原生 `open()` 函数，与 PHP 的 `include/require` 机制存在根本差异。以下为所有主流伪协议的兼容性分析：

| 协议 | PHP `include` | Python `open()` | 本系统是否受影响 | 分析 |
|:----:|:-------------:|:---------------:|:--------------:|------|
| `file://` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | `open("file:///etc/passwd")` 在 Python 中会尝试打开名为 `file:///etc/passwd` 的文件，不存在则报错 |
| `data://` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | `open("data://text/plain;base64,PD9waHA...")` Python 不识别 data URI scheme |
| `php://filter` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | `php://` 为 PHP 内置协议包装器，Python 无对应实现 |
| `http://` / `https://` | ✅ 支持（RFI） | ❌ 不支持 | ❌ 不受影响 | `open("http://attacker/shell.txt")` Python 抛出 `FileNotFoundError` |
| `ftp://` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | Python `open()` 不支持 FTP 协议 |
| `expect://` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | 仅 PHP expect 扩展支持 |
| `zip://` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | `zip://archive.zip#file.txt` Python 不识别 |
| `phar://` | ✅ 支持 | ❌ 不支持 | ❌ 不受影响 | PHP 特有的 Phar 协议 |

### 6.4 Python Flask 环境伪协议加固配置

针对本系统的实际 Flask 环境，推荐以下纵深配置：

```python
# [扩展加固] 在 app.py 中添加 URL scheme 校验拦截层（可选增强）
from urllib.parse import urlparse

SCHEME_BLACKLIST = {"file", "data", "http", "https", "ftp", "php", "expect", "zip", "phar", "gopher"}

def is_safe_path(name):
    """检测名称是否包含伪协议或不可信 scheme"""
    # 检查是否包含 :// （协议标识）
    if "://" in name:
        scheme = name.split("://")[0].lower()
        if scheme in SCHEME_BLACKLIST:
            return False
    # 检查常见伪协议前缀
    for scheme in SCHEME_BLACKLIST:
        if name.lower().startswith(scheme + ":"):
            return False
    return True

# 在 /page 路由中使用（目前已通过白名单拦截，此函数作为纵深防御层）
```

> **结论**：本系统为 Python Flask 架构，PHP 伪协议均不适用。当前的白名单方案已从根本上杜绝了 LFI/RFI 攻击面。上述伪协议分析作为审计完整性参考，同时为非 Python 环境迁移提供加固指引。

### 10.2 核心设计原则

本次修复遵循安全开发最佳实践：

| 原则 | 实施方式 | 涉及漏洞 |
|------|---------|:-------:|
| **白名单 > 黑名单** | 仅允许预设页面名 | 漏洞 1/2/3/4/5 |
| **纵深防御** | 白名单 + realpath + bleach 三重防护 | 漏洞 1/3/4/6 |
| **最小权限** | 输入不信任、文件名不保留 | 漏洞 8/9 |

| **防御分层** | 认证层 + 校验层 + 输出层 | 漏洞 7/1/6 |

### 10.3 持续改进建议

1. **非 root 运行**：建议使用非特权用户运行 Flask 进程，降低 SSH 私钥泄露风险
2. **HTTPS 部署**：在生产环境配置 TLS 证书，防止中间人攻击
3. **WAF 规则**：添加 `../`、`%2e%2e%2f` 等路径穿越特征的拦截规则
4. **日志监控**：对异常的 404 路径请求进行告警（高频目录穿越请求）
5. **密钥轮换**：定期更新 SSH 密钥对，缩短泄露影响窗口
