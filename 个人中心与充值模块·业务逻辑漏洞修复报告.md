## 二、漏洞详情与修复方案

---

### 【漏洞 1】/profile 接口水平越权查看他人隐私（IDOR）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /profile?user_id={id}` |
| **风险等级** | ⚠️ 高危 |
| **CWE** | CWE-639: Authorization Bypass Through User-Controlled Key |
| **风险判定依据** | 攻击者仅需修改 URL 中的 user_id 参数即可读取任意用户的手机号、邮箱、余额等敏感隐私信息，无需任何认证或授权；数据泄露范围覆盖整个用户库，影响全量用户；利用成本极低，仅需浏览器地址栏操作即可完成。符合《OWASP Top 10 2021》A01 失效访问控制的典型场景，综合评定为**高危**。 |

#### 形成原理

`/profile` 路由从 URL 参数接收 `user_id`，直接按 ID 查询用户并返回完整个人信息，**未做任何身份校验**——未判断当前登录用户是谁，也未判断当前用户是否有权限查看该 ID 对应的用户资料。

```python
@app.route("/profile")
def profile():
    user_id = request.args.get("user_id", type=int)   # ← 完全由客户端控制
    ...
    return render_template("profile.html", user=user)  # ← 返回完整隐私数据
```

#### 攻击利用方式

**场景 A — 直接查看任意用户**
```
GET /profile?user_id=1      → admin 的邮箱、手机、余额
GET /profile?user_id=2      → alice 的邮箱、手机、余额
```

**场景 B — 批量枚举**
```
for i in range(1, 1000):
    GET /profile?user_id={i}  → 遍历所有注册用户
```

#### 安全危害

- 所有用户的 **手机号、邮箱、余额** 等敏感信息批量泄露
- 攻击者可绘制用户信息图谱，用于社工攻击或撞库

#### 修复流程

1. 在 `/profile` 路由入口增加登录态检查，未登录用户重定向至登录页
2. 获取当前登录用户的 ID，校验其与请求的 `user_id` 是否匹配
3. 若不匹配则返回 HTTP 403，禁止越权访问
4. 对返回的手机号和邮箱字段做脱敏处理后渲染

#### 完整修复代码示例

```python
@app.route("/profile")
def profile():
    # 新增身份认证校验
    if not session.get("username"):
        return redirect("/login")

    login_username = session.get("username")
    login_user = USERS.get(login_username)
    if not login_user:
        return redirect("/login")

    user_id = request.args.get("user_id", type=int)
    if user_id is None:
        return "缺少 user_id 参数", 400

    # 校验当前登录用户只能查看自己的资料，防止水平越权
    if login_user["id"] != user_id:
        return "无权查看其他用户资料", 403

    # 从 USERS 字典中按 id 查找用户
    user = None
    for u in USERS.values():
        if u["id"] == user_id:
            user = dict(u)  # 复制一份以免修改原始数据
            break
    if user is None:
        return "用户不存在", 404

    # 敏感信息脱敏处理
    if user.get("phone") and len(user["phone"]) >= 7:
        user["phone"] = user["phone"][:3] + "****" + user["phone"][-4:]
    if user.get("email") and "@" in user["email"]:
        parts = user["email"].split("@")
        if len(parts[0]) >= 2:
            user["email"] = parts[0][0] + "***@" + parts[1]
        else:
            user["email"] = parts[0][0] + "@" + parts[1]

    return render_template("profile.html", user=user)
```

#### 复测方法

| 测试步骤 | 预期结果 | 修复结果 |
|---------|---------|:-------:|
| 未登录状态下访问 `/profile?user_id=1` | 重定向至 `/login` | ✅ 已阻断 |
| 以 Alice 身份登录后访问 `/profile?user_id=1` | 返回 403 无权访问 | ✅ 已阻断 |
| 以 Alice 身份登录后访问 `/profile?user_id=2` | 正常显示 Alice 本人资料（手机/邮箱已脱敏） | ✅ 已修复 |

### 【漏洞 2】/recharge 接口水平越权篡改他人余额（IDOR）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `POST /recharge` |
| **风险等级** | ⚠️ 高危 |
| **CWE** | CWE-639: Authorization Bypass Through User-Controlled Key |
| **风险判定依据** | 攻击者仅需拦截充值 POST 请求并修改 user_id 参数，即可对系统中任意账户执行余额增减操作。该漏洞直接影响资金安全，可导致系统内的虚拟资产被任意篡改，造成实质性的资金损失或账目混乱。与漏洞 4（无正数校验）组合可实现"给自己加钱、给别人扣钱"的完整攻击闭环。符合 CWE-639 高危权限绕过场景，综合评定为**高危**。 |

#### 形成原理

`/recharge` 路由从 POST 表单参数获取 `user_id`，**未校验该 user_id 是否与登录 session 中的用户匹配**，也未校验当前用户是否为管理员。攻击者只需抓包修改 `user_id` 即可操作任意账户余额。

```python
@app.route("/recharge", methods=["POST"])
def recharge():
    user_id = request.form.get("user_id", type=int)   # ← 完全由客户端控制
    amount = request.form.get("amount", type=float)   # ← 无正负限制
    ...
    u["balance"] += amount  # ← 直接累加
```

#### 攻击利用方式

**场景 A — 篡改管理员余额**
```
原始请求：user_id=2&amount=100         （Alice 给自己充值 ¥100）
篡改请求：user_id=1&amount=-50000      （Alice 将 admin 账户余额扣减 ¥50000）
```

**场景 B — 恶意扣减他人资金**
```
POST /recharge
user_id=2&amount=-10000            （将 Alice 余额扣为负数）
```

#### 安全危害

- 任意用户可 **篡改系统中所有账户的余额**
- 可造成系统资金账目混乱、虚拟资产损失

#### 修复流程

1. 在 `/recharge` 路由入口增加登录态检查
2. 将 `user_id` 的取值来源从前端表单/参数改为从 session 中获取，彻底杜绝越权篡改
3. 移除 `profile.html` 中的隐藏 `user_id` 字段（已不再需要）

#### 完整修复代码示例

```python
@app.route("/recharge", methods=["POST"])
def recharge():
    # 新增身份认证校验
    if not session.get("username"):
        return redirect("/login")

    login_username = session.get("username")
    login_user = USERS.get(login_username)
    if not login_user:
        return redirect("/login")

    # user_id 改为从 session 获取，不从表单参数读取，防止越权篡改他人余额
    user_id = login_user["id"]
    amount = request.form.get("amount", type=float)

    if amount is None:
        return "缺少金额参数", 400

    # 校验充值金额必须为正数
    if amount <= 0:
        return "充值金额必须为正数", 400

    # 单次充值金额上限限制
    MAX_AMOUNT = 100000
    if amount > MAX_AMOUNT:
        return f"单次充值金额不能超过 {MAX_AMOUNT}", 400

    # 使用 Decimal 精准计算，避免浮点精度误差
    amount = float(Decimal(str(amount)).quantize(Decimal("0.01"), rounding=ROUND_HALF_UP))

    # 按 id 查找用户并修改余额
    for u in USERS.values():
        if u["id"] == user_id:
            u["balance"] += amount
            break
    else:
        return "用户不存在", 404
    return redirect(f"/profile?user_id={user_id}")
```

#### 复测方法

| 测试步骤 | 预期结果 | 修复结果 |
|---------|---------|:-------:|
| 抓包篡改 `user_id` 参数（如 user_id=1）提交充值 | 后端忽略表单中的 user_id，仅从 session 获取 | ✅ 已阻断 |
| 未登录状态下直接 POST 调用 `/recharge` | 重定向至 `/login` | ✅ 已阻断 |

### 【漏洞 3】前端参数绕过——隐藏字段 user_id

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `POST /recharge`（结合 `templates/profile.html`） |
| **风险等级** | ⚠️ 中危 |
| **CWE** | CWE-602: Client-Side Enforcement of Server-Side Security |
| **风险判定依据** | 该漏洞本身不直接造成资金损失，但其为漏洞 2（越权篡改余额）提供了攻击入口——攻击者正是通过篡改前端隐藏字段的值来实现越权操作。该漏洞的风险实现依赖漏洞 2、4、8 的配合触发，属于支撑性风险而非独立高危入口；但其反映出"前端限制即安全"的认知缺陷，在纵深防御层面具有重要意义。综合评定为**中危**。 |

#### 形成原理

`profile.html` 使用隐藏 `<input>` 传递 `user_id`：

```html
<input type="hidden" name="user_id" value="{{ user.id }}">
```

这种"隐藏即安全"的认知是典型误区。隐藏字段对用户不可见，但 **对 HTTP 代理（Burp Suite、浏览器 DevTools）完全透明**，攻击者可轻易拦截并篡改。

| 前端限制手段 | 绕过方式 |
|-------------|---------|
| `<input type="hidden">` | 浏览器 F12 修改 value 属性 |
| 页面只显示当前用户 | 直接构造 `curl -X POST -d "user_id=1&amount=..."` 请求 |
| JS 前端校验 | 禁用 JS 或直接发原始 HTTP 请求 |

#### 攻击利用方式

```
# 浏览器看到：当前用户为 Alice (user_id=2)
# 攻击者拦截请求后修改：
POST /recharge
Body: user_id=1&amount=99999  → 给 admin 充值

POST /recharge
Body: user_id=1&amount=-99999 → 清空 admin 余额
```

#### 安全危害

- 隐藏字段仅提供"防手滑"级别的安全，对恶意攻击者无效
- 配合漏洞 1（已获知他人 user_id），可对任意账户实施资金操作

#### 修复流程

1. 移除 `profile.html` 中隐藏的 `<input name="user_id">` 字段
2. 在服务端将 `user_id` 的取值来源从前端表单改为 session 中已登录用户的 ID
3. 确保充值操作的目标用户不可由前端指定

#### 完整修复代码示例

**profile.html** —— 移除隐藏字段后的充值表单：

```html
<form method="post" action="/recharge" class="recharge-form">
    <div class="form-group">
        <label for="amount">充值金额：</label>
        <input type="number" id="amount" name="amount"
               class="form-input" step="0.01" required>
    </div>
    <button type="submit" class="btn btn-primary">充值</button>
</form>
```

**app.py** —— 服务端从 session 获取 user_id（与漏洞 2 共用同一修复代码）：

```python
user_id = login_user["id"]  # 从 session 获取，而非 request.form
```

#### 复测方法

| 测试步骤 | 预期结果 | 修复结果 |
|---------|---------|:-------:|
| 检查 profile.html 是否存在 `<input name="user_id">` | 已移除，页面无此字段 | ✅ 已修复 |
| 以 Alice 登录后查看充值请求 Body | 请求中不包含 user_id 参数 | ✅ 已修复 |

### 【漏洞 4】充值金额无正数校验（可直接扣减余额）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `POST /recharge` |
| **风险等级** | ⚠️ 高危 |
| **CWE** | CWE-841: Improper Enforcement of Behavioral Workflow |
| **风险判定依据** | amount 参数为负数时，"充值"操作直接变为"扣款"行为，完全违背业务语义。该漏洞与漏洞 2（越权）组合可构成完整的资金窃取攻击链——攻击者通过修改 user_id 将他人账户作为扣款目标、通过负数 amount 实现余额扣减。该漏洞独立存在时影响可控（仅能操作自己的余额），但与越权漏洞组合后升级为资金窃取能力。由于其作为资金窃取链中不可缺失的关键环节，综合评定为**高危**。 |

#### 形成原理

第 312 行无条件执行 `balance = balance + amount`，**未校验 amount 是否为正数**：

```python
amount = request.form.get("amount", type=float)   # 未校验正负
u["balance"] += amount                              # 负值即为扣款
```

| 传入值 | 业务结果 | 是否符合预期 |
|--------|---------|:----------:|
| `amount=100` | 余额 +100 ✅ | ✅ |
| `amount=-100` | 余额 -100 ❌ | **❌ 负数被接受** |
| `amount=0` | 余额不变 | **❌ 无意义操作被接受** |
| `amount=-99999` | 余额清零或变负 | **❌ 恶意扣款** |

#### 攻击利用方式

```
# 以 Alice 身份登录，对 admin 账户操作：
POST /recharge
user_id=1&amount=-50000    → admin 余额从 99999 降至 49999
POST /recharge
user_id=1&amount=-99999    → admin 余额归零
```

#### 安全危害

- 任意用户可通过 **负数充值** 直接扣减他人余额
- 系统资金逻辑被彻底破坏，余额可信度归零
- 若余额出现负数，可能导致下游系统异常（如无法提现但显示负债）

#### 修复流程

1. 在路由中获取 amount 后增加正数校验
2. 若 `amount <= 0` 则直接返回 400 错误并提示"充值金额必须为正数"

#### 完整修复代码示例

```python
amount = request.form.get("amount", type=float)

if amount is None:
    return "缺少金额参数", 400

# 校验充值金额必须为正数
if amount <= 0:
    return "充值金额必须为正数", 400
```

#### 复测方法

| 测试步骤 | 预期结果 | 修复结果 |
|---------|---------|:-------:|
| 充值金额传 `-100` | 返回 400 "充值金额必须为正数" | ✅ 已阻断 |
| 充值金额传 `0` | 返回 400 "充值金额必须为正数" | ✅ 已阻断 |
| 充值金额传 `99.99` | 正常充值成功，余额增加 | ✅ 已通过 |

### 【漏洞 5】充值金额无数量级限制

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `POST /recharge` |
| **风险等级** | ⚠️ 中危 |
| **CWE** | CWE-770: Allocation of Resources Without Limits or Throttling |
| **风险判定依据** | 无上限限制可导致单次充值金额异常庞大（如 `amount=1e20`），引发数据溢出或浮点精度丢失；无下限约束配合负数漏洞可无限精确扣减。但由于系统当前为内存存储、未对接真实支付网关，大额充值风险主要体现在数据层面而非实际资金损失；且该漏洞独立存在时不会导致信息泄露或越权，仅为数值边界缺失。综合评定为**中危**。 |

#### 形成原理

amount 为 `float` 类型，无上限约束，也无极小值限制：

```python
amount = request.form.get("amount", type=float)   # float 类型，接受任意值
```

#### 攻击利用方式

```
# 场景 A — 超大额充值
POST /recharge
user_id=1&amount=999999999999999999  → 余额暴涨

# 场景 B — 极小值
POST /recharge
user_id=1&amount=0.000000001         → 极小值操作（虽无害但无意义）

# 场景 C — 浮点精度问题
POST /recharge
user_id=1&amount=0.1                → 浮点累加产生精度误差
```

#### 安全危害

- 大额操作可导致数据溢出或显示异常
- 浮点累积误差导致余额精度失真
- 缺乏变更记录，资金操作不可追溯

#### 修复流程

1. 设定合理的单次充值上限（如 100000）
2. 使用 `Decimal` 类型替代 `float` 进行金额计算，消除浮点精度误差
3. 统一金额精度为两位小数

#### 完整修复代码示例

```python
from decimal import Decimal, ROUND_HALF_UP

# 单次充值金额上限限制
MAX_AMOUNT = 100000
if amount > MAX_AMOUNT:
    return f"单次充值金额不能超过 {MAX_AMOUNT}", 400

# 使用 Decimal 精准计算，避免浮点精度误差
amount = float(Decimal(str(amount)).quantize(
    Decimal("0.01"), rounding=ROUND_HALF_UP
))
```

#### 复测方法

| 测试步骤 | 预期结果 | 修复结果 |
|---------|---------|:-------:|
| 充值金额传 `999999999`（超上限） | 返回 400 "单次充值金额不能超过 100000" | ✅ 已阻断 |
| 充值金额传 `0.1`，执行 3 次后查看余额 | 余额精确为 `0.30`，无浮点误差 | ✅ 已修复 |

### 【漏洞 6】敏感信息未脱敏输出

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /profile` |
| **风险等级** | ⚠️ 中危 |
| **CWE** | CWE-200: Exposure of Sensitive Information to an Unauthorized Actor |
| **风险判定依据** | 手机号和邮箱以明文完整展示在页面上，一旦越权漏洞（漏洞 1）被利用，泄露的数据质量极高——攻击者获得的是直接可用的手机号和邮箱，无需额外破解。该漏洞本身不直接导致泄露（需要 IDOR 配合），但显著放大了 IDOR 漏洞的危害程度，属于信息泄露的"放大器"。综合评定为**中危**。 |

#### 形成原理

`profile.html` 直接将数据库字段原始值渲染到页面：

```html
<li><span class="info-label">邮　箱：</span>{{ user.email }}</li>
<li><span class="info-label">手　机：</span>{{ user.phone }}</li>
<li><span class="info-label">余　额：</span>{{ user.balance }}</li>
```

未做任何脱敏处理（如手机号 `138****8000`、邮箱 `a***@example.com`）。

#### 攻击利用方式

```
GET /profile?user_id=1

响应页面包含：
- 邮　箱：admin@example.com         ← 完整邮箱
- 手　机：13800138000               ← 完整手机号
- 余　额：99999                     ← 完整余额
```

#### 安全危害

- 手机号和邮箱完整暴露，可用于精准钓鱼或社工攻击
- 余额信息公开暴露用户财务状况，可能引发针对高余额账户的精准攻击

#### 修复流程

1. 在 `/profile` 路由中，查询到用户数据后在返回模板之前进行脱敏处理
2. 手机号保留前 3 位和后 4 位，中间用 `****` 代替（如 `138****8000`）
3. 邮箱保留首字符和 @ 后域名，中间用 `***` 代替（如 `a***@example.com`）

#### 完整修复代码示例

```python
# 敏感信息脱敏处理
if user.get("phone") and len(user["phone"]) >= 7:
    user["phone"] = user["phone"][:3] + "****" + user["phone"][-4:]
if user.get("email") and "@" in user["email"]:
    parts = user["email"].split("@")
    if len(parts[0]) >= 2:
        user["email"] = parts[0][0] + "***@" + parts[1]
    else:
        user["email"] = parts[0][0] + "@" + parts[1]
```

#### 复测方法

| 测试步骤 | 预期结果 | 修复结果 |
|---------|---------|:-------:|
| 登录 admin 后访问 `/profile?user_id=1` | 手机显示 `138****8000`，邮箱显示 `a***@example.com` | ✅ 已修复 |

### 【漏洞 7】无需登录即可访问 /profile

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /profile` |
| **风险等级** | ⚠️ 高危 |
| **CWE** | CWE-306: Missing Authentication for Critical Function |
| **风险判定依据** | `/profile` 接口未做任何登录态校验，攻击者无需任何 Cookie/Session 即可直接通过 URL 访问任意用户的个人资料。该漏洞使得漏洞 1（IDOR）的攻击面从"已登录用户越权"扩大为"无需任何身份即可越权"——攻击门槛降至零。同时该接口返回完整敏感信息，相当于将整个用户数据库暴露在公网之上。符合 CWE-306 关键功能缺乏认证的高危场景，综合评定为**高危**。 |

#### 形成原理

对比已有的其他路由：`/search` 和 `/upload` 都检查了 `session.get("username")`，但 `/profile` 路由没有：

```python
# /search 有身份校验
@app.route("/search")
def search():
    if not session.get("username"):
        return redirect("/login")     # ✅ 有校验

# /upload 有身份校验
@app.route("/upload")
def upload():
    if not session.get("username"):
        return redirect("/login")     # ✅ 有校验

# /profile 无身份校验  ❌
@app.route("/profile")
def profile():
    user_id = request.args.get("user_id", type=int)
    ...                               # ❌ 未检查登录态
```

#### 攻击利用方式

```
# 完全无需登录，甚至不需要 Cookie：
curl -s "http://target/profile?user_id=1"
# → 直接返回 admin 的完整资料

curl -s "http://target/profile?user_id=2"
# → 直接返回 alice 的完整资料
```

#### 安全危害

- **无需任何认证**即可访问所有用户的敏感信息
- 结合批量枚举，可构成大规模数据泄露事件

#### 修复流程

1. 在 `/profile` 路由最上方增加登录态校验
2. 若 `session` 中不存在 `username` 则重定向至登录页

#### 完整修复代码示例

```python
@app.route("/profile")
def profile():
    # 新增身份认证校验
    if not session.get("username"):
        return redirect("/login")

    login_username = session.get("username")
    login_user = USERS.get(login_username)
    if not login_user:
        return redirect("/login")
    # ... 后续业务逻辑
```

#### 复测方法

| 测试步骤 | 预期结果 | 修复结果 |
|---------|---------|:-------:|
| 清除浏览器 Cookie 后直接访问 `/profile?user_id=1` | 重定向至 `/login` | ✅ 已阻断 |
| 未携带 Session Cookie 的 curl 请求 | 返回 302 重定向 | ✅ 已阻断 |

### 【漏洞 8】无需登录即可调用 /recharge

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `POST /recharge` |
| **风险等级** | ⚠️ 高危 |
| **CWE** | CWE-306: Missing Authentication for Critical Function |
| **风险判定依据** | 资金操作接口（/recharge）完全未做身份认证，任何能够访问该服务的人均可发送 POST 请求操作任意账户余额。该漏洞是所有资金安全风险中最为根本的缺陷——即使修复了越权（漏洞 2）和金额检验（漏洞 4/5），只要接口开放给未认证用户，攻击者仍可通过自动化脚本大量调用产生拒绝服务或资源耗尽风险。资金操作接口缺失认证属于最严重的安全设计缺陷之一，综合评定为**高危**。 |

#### 形成原理

`/recharge` 路由同样没有任何身份校验：

```python
@app.route("/recharge", methods=["POST"])
def recharge():
    user_id = request.form.get("user_id", type=int)
    amount = request.form.get("amount", type=float)
    ...                               # ❌ 未检查登录态
```

#### 攻击利用方式

```
# 无需登录，无需 Cookie，一条 curl 即可操控任意账户余额：
curl -X POST "http://target/recharge" \
  -d "user_id=1&amount=-99999"
# → admin 余额清零

curl -X POST "http://target/recharge" \
  -d "user_id=2&amount=1000000"
# → alice 余额暴涨至百万
```

#### 安全危害

- 资金操作接口完全暴露在公网，**任何人都可以零成本发起攻击**
- 可与 /profile 配合（先枚举 user_id，再逐个操作余额）
- 系统资金数据完全失陷，造成实际经济损失

#### 修复流程

1. 在 `/recharge` 路由最上方增加登录态校验
2. 若 `session` 中不存在 `username` 则重定向至登录页

#### 完整修复代码示例

```python
@app.route("/recharge", methods=["POST"])
def recharge():
    # 新增身份认证校验
    if not session.get("username"):
        return redirect("/login")

    login_username = session.get("username")
    login_user = USERS.get(login_username)
    if not login_user:
        return redirect("/login")

    # user_id 改为从 session 获取
    user_id = login_user["id"]
    # ... 后续业务逻辑与金额校验
```

#### 复测方法

| 测试步骤 | 预期结果 | 修复结果 |
|---------|---------|:-------:|
| 未登录状态下 curl POST `/recharge` | 重定向至 `/login`，余额不变 | ✅ 已阻断 |

## 三、攻击链串联分析

以上漏洞可组合实现 **从信息采集到资金窃取的自动化攻击链**：

```
┌─────────────────────────────────────────────────────┐
│              自动化攻击链（无需登录）                     │
├─────────────────────────────────────────────────────┤
│                                                      │
│  第1步：信息收集                                      │
│  利用 漏洞1+漏洞7（IDOR + 无认证）                     │
│  for id in range(1, 1000):                           │
│      GET /profile?user_id={id}                       │
│      → 获取所有用户的 ID、手机号、邮箱、余额             │
│                                                      │
│  第2步：筛选高价值目标                                  │
│  余额 > 0 的用户中筛选：                               │
│  → admin (id=1, 余额=99999)                          │
│  → alice (id=2, 余额=100)                            │
│                                                      │
│  第3步：资金操作                                      │
│  利用 漏洞2+漏洞4+漏洞8（越权 + 负数金额 + 无认证）      │
│                                                      │
│  扣减高价值目标余额（操作A）     增加自有余额（操作B）   │
│  POST /recharge                     POST /recharge    │
│  Body: user_id=1&amount=-99999     Body: user_id=3    │
│  → admin 余额归零                    → 余额 +99999    │
│                                                      │
│  注：本系统无转账功能，攻击者通过扣减他人加自己模拟资金转移 │
└─────────────────────────────────────────────────────┘
```

**攻击成本评估**：整套攻击链仅需 `curl` 命令，**无需任何认证凭据**，耗时数秒即可完成从信息收集到资金清空的全流程操作。

**修复关键阻断点分析**：

| 攻击步骤 | 可利用的漏洞 | 修复后阻断点 | 修复有效性 |
|---------|-------------|-------------|:---------:|
| 信息收集 | 漏洞 1 + 漏洞 7 | 增加登录校验 + IDOR 检查 | **❖ 完全阻断** |
| 资金操作（越权） | 漏洞 2 + 漏洞 3 | user_id 改从 session 获取 | **❖ 完全阻断** |
| 资金操作（负数） | 漏洞 4 | amount ≤0 时拒绝 | **❖ 完全阻断** |
| 资金操作（无认证） | 漏洞 8 | 增加登录校验 | **❖ 完全阻断** |

修复后攻击链在第 1 步即被阻断——未登录状态下无法访问 `/profile`，也无法调用 `/recharge`，后续所有攻击步骤无从展开。
